四、异常 / 失败分支

当前代码已处理的主要异常

登录态失效

小程序请求层遇到 `401` 会清掉本地 session，再次调用 `ensureSession()` 重新走微信登录。

旧草稿单失效

结算页若支付意图阶段收到 `Pending order not found.`，会清本地 draft order id，重新创建草稿单再发起支付。

支付状态未及时完成

支付结果页会轮询订单；若暂时没进入可跳转状态，会提示用户稍后查看订单，不会直接丢失订单入口。

现状边界

  • 后台 `admin` 已经全面走真实 `/admin/*` 接口。
  • 小程序也已经有真实请求层和接口路径。
  • 但默认开发态开启了 `TARO_APP_SHOWCASE_DATA=true`，所以首页、详情、订单、我的、观演人等页面会优先走展示数据。

为什么研发会觉得“没打通”

因为默认开发态打开后，页面层很多地方会被 showcase 数据短路，用户肉眼看到的是“页面能动”，但不一定真的正在从后端取实时数据。 所以更准确的说法是：请求层已写、接口已接，但默认运行态不是完整真实联调态。

切成真实联调模式要关的开关

  • `TARO_APP_SHOWCASE_DATA=false`
  • `TARO_APP_MOCK_WECHAT_PAY=false`
  • 补真实 `WECHAT_APP_ID / WECHAT_APP_SECRET / 支付参数`
  • 确保 API 地址指向真实可访问环境

适合给团队对齐的结论

这套小程序不是“完全没联接口”，而是“代码层已接接口，但默认开发态保留了 showcase/mock 便利能力”。 交接和联调时要先分清“开发演示流”和“真实业务流”。
