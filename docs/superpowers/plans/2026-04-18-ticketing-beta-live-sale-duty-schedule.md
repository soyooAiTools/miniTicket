# 票务内测首场开卖值守时间表

## 目的

这份时间表把内测 runbook 变成一张按时间推进的开卖日执行单。

用在首场真实公开售卖当天，帮助一个小团队明确谁在什么时间看什么、什么时候切状态、什么时候该暂停流量继续放大。

- 总指挥：
- 后端值守：
- 运营值守：

- 上游联系人：
- 内部升级通道：

## T-60 分钟

### 总指挥

- [ ] 拉起指挥通道。
- [ ] 确认 4 个角色负责人全部在线。
- [ ] 确认目标演出和开售时间。
- [ ] 确认当前上游沟通通道可用。

### 后端值守

- [ ] 确认预发 / 线上 API 地址可访问。
- [ ] 确认部署日志可访问。
- [ ] 确认支付回调和上游回调日志都能查看。

### 运营值守

- [ ] 打开后台 `/events`。
- [ ] 打开后台 `/orders`。
- [ ] 打开后台 `/fulfillment`。
- [ ] 打开后台 `/refunds`。
- [ ] 确认演出的 `published`、`saleStatus`、`refundEntryEnabled` 状态符合预期。

### 客服值守

- [ ] 打开已审核的话术表。
- [ ] 准备好待出票、出票失败、退款中、退款失败、身份信息错误扣费的话术。
- [ ] 确认异常单升级标签或记录方式。

## T-30 分钟

### 后端值守

- [ ] 轻量检查 `GET /api/catalog/events`。
- [ ] 再跑一次新登录，确认仍成功。
- [ ] 确认当前没有已经积累的部署错误。

### 运营值守

- [ ] 确认小程序演出详情可打开。
- [ ] 确认后台 `/orders` 没有遗留积压。
- [ ] 确认后台 `/fulfillment` 没有遗留积压。
- [ ] 确认后台 `/refunds` 没有遗留积压。

### 总指挥

- [ ] 审看 smoke check 结果。
- [ ] 确认这次内测窗口只有目标演出对外公开。

## T-10 分钟

### 总指挥

- [ ] 确认 `published=true`。
- [ ] 如果现在要开始售卖，确认 `saleStatus=ON_SALE`。
- [ ] 确认 `refundEntryEnabled` 符合用户侧策略。
- [ ] 确认 4 个角色负责人仍然在线。

### 客服值守

- [ ] 发出“开卖窗口开启”的内部通知。
- [ ] 随时准备接“支付成功但未出票”的用户问题。

## T0 到 T+15 分钟

### 监控重点

- [ ] 观察小程序浏览、下单、支付反馈。
- [ ] 观察后台 `/orders` 中新产生的已支付单。
- [ ] 观察后台 `/fulfillment` 中已支付未出票单。
- [ ] 观察微信支付回调相关日志。
- [ ] 观察上游提交失败相关日志。

### 通报节奏

- [ ] 运营值守每 `10` 分钟播报一次状态。
- [ ] 后端值守每 `10` 分钟播报一次回调健康情况。
- [ ] 总指挥记录当前流量是否稳定。

## T+15 到 T+60 分钟

### 稳定标志

- [ ] 新用户可以登录。
- [ ] 新草稿单可以创建。
- [ ] 支付可以完成。
- [ ] 支付成功后可以进入履约处理。
- [ ] 至少有一笔订单进入用户可见“已出票”。
- [ ] 后台页面持续可读、持续刷新正常。

### 升级触发条件

- [ ] 任意已支付订单在支付回调成功后超过 `15` 分钟仍无出票进展。
- [ ] 微信支付回调对多笔订单连续失败。
- [ ] 上游回调延迟导致待出票队列持续增长。
- [ ] 退款请求在创建或回调阶段开始失败。
- [ ] 后台状态和小程序订单详情状态出现明显漂移。

### 触发后的立即动作

- [ ] 总指挥暂停继续扩大流量。
- [ ] 后端值守针对受影响订单检查支付和上游日志。
- [ ] 运营值守记录当前各队列数量。
- [ ] 客服值守把相关用户切到异常单回复路径。

## T+90 分钟复盘点

- [ ] 统计未解决客服 case 数。
- [ ] 决定保持、缩减还是暂停当前流量暴露。

## 应急控制说明

遇到严重事故时，总指挥可以：

- [ ] 设 `published=false`，让演出从公开浏览入口消失。
- [ ] 修改 `saleStatus`，改变用户可见的销售状态。
- [ ] 设 `refundEntryEnabled=false`，隐藏用户侧退款入口。

但由于 `saleStatus` 和 `refundEntryEnabled` 还不是所有写路径的后端硬熔断，还必须同步完成：

- [ ] 停止新的推广流量导入。
- [ ] 持续盯住在途支付回调。
- [ ] 持续盯住在途上游回调。
- [ ] 给每一笔受影响订单补齐明确负责人。
