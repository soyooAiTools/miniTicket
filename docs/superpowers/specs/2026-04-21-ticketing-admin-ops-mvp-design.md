# 票务后台运营管理系统 MVP 设计

## 1. 背景

当前 `miniTicket` 已有一版基础后台，包含活动、订单、履约、退款四个页面，但整体仍偏向“基础数据表格后台”：

- 缺少正式后台登录和角色权限
- 活动页仅支持部分状态修改，无法完成真正的活动创建和上架
- 订单、履约、退款以只读展示为主，缺少运营处理动作
- 信息架构偏散，履约单独顶层暴露，不利于运营快速判断“该去哪处理事情”
- 现有票务数据更接近“票档价格”模型，尚未明确支持大型演出的“区域票”

本次设计目标是把现有后台升级为“真可运营 MVP”，服务于内部运营团队日常工作，并与当前小程序端的票务主链路保持一致。

## 2. 目标与非目标

### 2.1 目标

第一版后台需要同时满足两类诉求：

1. 具备真实可执行的运营能力，而不是只展示 mock 数据
2. 具备完整、专业、中文化的后台工作台体验

本次 MVP 的明确目标如下：

- 支持后台正式登录和基础角色权限
- 提供运营概览页，帮助团队快速定位待处理事项
- 支持活动创建、编辑、区域票档配置、保存草稿、发布、下架
- 支持订单查询、订单详情、内部备注、异常单标记
- 支持退款审核、驳回、发起退款、处理备注
- 支持最小账号管理：创建账号、启停账号、分配角色
- 对关键后台动作保留操作日志

### 2.2 非目标

以下内容不在本次 MVP 范围内：

- 真实座位图选座
- 细粒度 RBAC 权限树
- 财务对账、结算、报表中心
- 分销、邀请码、渠道体系
- 复杂审批流
- 后台手工篡改支付结果或出票结果
- 后台直接人工补票码、强制改订单核心状态

## 3. 设计原则

### 3.1 产品原则

- 后台是“运营工作台”，不是“功能展示台”
- 页面只展示真正需要处理的信息，不堆解释型文案
- 中文优先，避免中英混排的后台感
- 高风险动作严格收口，避免后台成为“万能改数据入口”

### 3.2 实现原则

- 尽量复用当前 `apps/admin`、`apps/api` 与 Prisma 结构，不做无必要推翻
- 优先打通“活动上架主流程”，同时让订单与退款能接住真实问题
- 履约数据继续保留，但不再作为一级导航模块暴露
- 通过新增后台专用接口与后台登录态，把管理端与小程序用户端能力清晰隔离

## 4. 当前基础与改造方向

### 4.1 当前已有基础

仓库中已有以下后台页面能力：

- `活动`：可查询后台活动列表，并修改部分发布与售卖状态
- `订单`：可检索与查看后台订单列表
- `履约`：可查看履约事件与上游引用
- `退款`：可查看退款申请列表

### 4.2 本次改造方向

保留现有 React + Ant Design 的后台基础，但将其升级为真正的运营工作台：

- 新增统一后台登录入口与登录态
- 重构后台导航与页面布局
- 让 `活动` 成为后台第一优先主流程
- 将 `履约` 融入订单详情，而非单独顶层页面
- 为订单、退款提供真实处理动作与日志留痕

## 5. 信息架构

后台 MVP 的一级导航固定为：

- `概览`
- `活动`
- `订单`
- `退款`
- `账号`

补充页面：

- `登录`

不再保留一级导航：

- `履约`

履约信息会保留为订单详情中的一个区块，用于运营排查出票状态与上游履约记录。

## 6. 角色与权限

### 6.1 角色

第一版后台只启用两类角色：

- `超级管理员`
- `运营`

与当前 Prisma 中的 `UserRole` 对应关系如下：

- `ADMIN` -> 超级管理员
- `OPERATIONS` -> 运营

### 6.2 权限边界

#### 超级管理员

- 登录后台
- 访问概览、活动、订单、退款、账号
- 创建/编辑/发布/下架活动
- 配置区域票档
- 查看和处理订单、退款
- 创建账号、禁用账号、变更角色

#### 运营

- 登录后台
- 访问概览、活动、订单、退款
- 创建/编辑/发布/下架活动
- 配置区域票档
- 查看订单、添加备注、标记异常
- 审核退款、驳回退款、发起退款
- 不可进入账号管理

### 6.3 权限非目标

第一版不提供：

- 页面级自由配置权限
- 字段级权限
- 数据范围隔离

## 7. 核心业务模型

## 7.1 活动模型

沿用现有主结构：

- `Event`
- `EventSession`
- `TicketTier`

但将 `TicketTier` 在业务语义上升级为“区域票档”。

### 7.2 区域票档模型

第一版采用“区域票”，不是“座位图选座”。

用户购票时只选择区域，不选择具体座位号。

一个场次下可配置多个区域票档，例如：

- 内场 A 区：1280
- 内场 B 区：980
- 看台一区：680
- 看台二区：480

每个区域票档至少包含以下字段：

- 区域名称
- 售价
- 库存
- 每单限购
- 是否实名
- 是否允许退款
- 退款截止时间
- 票种类型
- 展示排序

建议在 Prisma `TicketTier` 上补充以下字段：

- `purchaseLimit Int`
- `requiresRealName Boolean`
- `refundable Boolean`
- `refundDeadlineAt DateTime?`
- `sortOrder Int`

`name` 字段承载区域名称，例如“内场 A 区”。

### 7.3 订单运营模型

为支持运营处理，需要补充后台运营数据：

- 订单内部备注
- 订单异常标记
- 操作日志

建议新增：

- `OrderNote`
  - `id`
  - `orderId`
  - `content`
  - `createdByUserId`
  - `createdAt`

- `OrderFlag`
  - `id`
  - `orderId`
  - `type`
  - `note`
  - `createdByUserId`
  - `createdAt`

其中 `type` 第一版建议固定支持：

- `异常单`
- `待人工核查`
- `用户争议`

### 7.4 退款运营模型

在现有 `RefundRequest` 基础上补充后台处理字段：

- `reviewedByUserId`
- `reviewNote`
- `rejectionReason`
- `processedByUserId`
- `lastHandledAt`

这些字段用于支撑审核、驳回、退款处理与后续追溯。

### 7.5 后台会话与审计模型

建议新增：

- `AdminSession`
  - `id`
  - `userId`
  - `tokenHash`
  - `expiresAt`
  - `createdAt`
  - `updatedAt`

- `AdminAuditLog`
  - `id`
  - `userId`
  - `action`
  - `targetType`
  - `targetId`
  - `payload`
  - `createdAt`

`AdminAuditLog` 用于记录关键后台动作：

- 登录/登出
- 创建活动
- 编辑活动
- 发布/下架活动
- 添加订单备注
- 标记异常单
- 审核/驳回/发起退款
- 创建/禁用账号

## 8. 核心运营流程

### 8.1 活动上架主流程

这是本次 MVP 的第一优先主流程。

流程如下：

1. 运营进入活动列表
2. 点击“新建活动”
3. 填写活动基本信息
4. 配置场次
5. 配置区域票档
6. 配置销售规则与售后规则
7. 保存为草稿，或直接发布
8. 后续支持再次编辑或下架

活动状态线固定为：

- `草稿`
- `待开售`
- `售卖中`
- `已下架`

说明：

- 保存草稿不对小程序端可见
- 发布后根据开售时间进入 `待开售` 或 `售卖中`
- 下架后对小程序端不可购买

### 8.2 订单中心处理流

订单中心第一版定位为“问题定位与跟进处理”，而非高风险人工改单后台。

流程如下：

1. 运营通过订单号、用户、活动、状态等条件查单
2. 查看订单详情、支付状态、观演人信息、区域票档、履约记录、退款记录
3. 必要时添加内部备注
4. 必要时标记异常单
5. 若涉及退款，跳转退款中心处理

明确限制：

- 不允许后台直接改支付成功/失败
- 不允许后台手工改出票成功/失败
- 不允许后台手工切换订单核心状态

### 8.3 退款处理流

退款中心第一版做成标准处理队列。

状态线固定为：

- `待审核`
- `已通过`
- `已驳回`
- `处理中`
- `已完成`

处理流程如下：

1. 运营进入退款列表
2. 优先查看待审核申请
3. 打开退款详情
4. 审核通过或驳回
5. 通过后可发起退款处理
6. 记录处理备注和操作日志

## 9. 页面与交互设计

### 9.1 整体布局

后台采用：

- 左侧导航
- 顶部栏
- 主内容区

设计要求：

- 全中文
- 浅色底、冷白灰主色
- 页面以卡片化区块承载信息
- 状态标签统一
- 不出现解释型大段文案

### 9.2 登录页

登录页仅提供：

- 邮箱
- 密码
- 登录按钮

不在第一版做：

- 找回密码
- 双因素认证

### 9.3 概览页

概览页展示运营待办，而不是复杂图表。

核心区块：

- 待处理退款数
- 异常订单数
- 即将开售活动数
- 当前售卖中活动数
- 最近操作记录

### 9.4 活动列表页

功能：

- 搜索
- 按状态筛选
- 按城市筛选
- 创建活动
- 查看活动状态、最低价、最近更新时间

列表页是活动主流程入口，不承载复杂编辑表单。

### 9.5 活动编辑页

采用分区式表单，不做一整页长表单堆叠。

固定区块：

- 基本信息
  - 标题
  - 副标题
  - 海报
  - 城市
  - 场馆名称
  - 场馆地址
  - 活动介绍

- 场次配置
  - 场次名称
  - 开始时间
  - 结束时间
  - 开售时间
  - 停售时间

- 区域票档配置
  - 区域名称
  - 售价
  - 库存
  - 每单限购
  - 是否实名
  - 是否可退
  - 退款截止时间
  - 票种类型
  - 排序

- 销售规则
  - 是否实名
  - 限购规则
  - 开售时间
  - 停售时间

- 售后规则
  - 是否允许退款
  - 退款截止时间
  - 退款说明

页面底部操作固定为：

- `保存草稿`
- `发布活动`
- `下架活动`

### 9.6 订单列表与详情页

订单列表页：

- 顶部搜索和筛选
- 列表显示订单号、活动、区域票档、支付状态、履约状态、退款状态、更新时间

订单详情页：

- 订单基础信息
- 用户与观演人信息
- 区域票档信息
- 支付记录
- 履约记录
- 退款记录
- 内部备注时间线
- 异常标记

右侧固定“运营操作栏”：

- 添加备注
- 标记异常单
- 跳转退款中心

### 9.7 退款列表与详情页

退款列表页：

- 默认按待处理优先排序
- 支持状态筛选、订单号筛选、活动筛选

退款详情页：

- 订单信息
- 用户信息
- 退款原因
- 金额拆分
- 审核记录
- 处理备注

固定操作区：

- 审核通过
- 驳回
- 发起退款
- 添加备注

### 9.8 账号页

账号页第一版功能固定为：

- 账号列表
- 创建账号
- 启用/停用账号
- 角色切换

不提供复杂权限树。

## 10. 接口设计

后台接口统一使用 `/admin` 前缀，与小程序用户接口隔离。

### 10.1 认证接口

- `POST /admin/auth/login`
- `POST /admin/auth/logout`
- `GET /admin/auth/me`

### 10.2 概览接口

- `GET /admin/dashboard/summary`

返回内容包含：

- 待处理退款数量
- 异常订单数量
- 即将开售活动数量
- 售卖中活动数量
- 最近操作记录

### 10.3 活动接口

- `GET /admin/events`
- `POST /admin/events`
- `GET /admin/events/:id`
- `PATCH /admin/events/:id`
- `POST /admin/events/:id/publish`
- `POST /admin/events/:id/unpublish`

活动创建和编辑接口一次性提交：

- 活动基本信息
- 场次配置
- 区域票档配置
- 销售规则
- 售后规则

### 10.4 订单接口

- `GET /admin/orders`
- `GET /admin/orders/:id`
- `POST /admin/orders/:id/notes`
- `POST /admin/orders/:id/flags`

### 10.5 退款接口

- `GET /admin/refunds`
- `GET /admin/refunds/:id`
- `POST /admin/refunds/:id/approve`
- `POST /admin/refunds/:id/reject`
- `POST /admin/refunds/:id/process`
- `POST /admin/refunds/:id/notes`

### 10.6 账号接口

- `GET /admin/users`
- `POST /admin/users`
- `PATCH /admin/users/:id`
- `POST /admin/users/:id/enable`
- `POST /admin/users/:id/disable`

## 11. 表单与数据校验

### 11.1 活动校验

- 标题必填
- 城市必填
- 场馆名称必填
- 至少存在一个场次
- 每个场次至少存在一个区域票档
- 票档价格必须大于 0
- 库存必须大于等于 0
- 每单限购必须大于 0
- 同一场次下区域票档名称不可重复
- 退款截止时间不得晚于活动开始时间
- 开售时间不得晚于停售时间

### 11.2 发布校验

活动满足以下条件才允许发布：

- 基本信息完整
- 至少一个场次完整
- 至少一个区域票档完整
- 关键销售规则完整

### 11.3 退款校验

- 驳回必须填写驳回原因
- 发起退款金额不得超过可退金额
- 已完成退款不可重复处理

## 12. 安全与风控边界

### 12.1 登录安全

后台采用服务端 session + 安全 cookie，不使用前端长期存储密钥作为正式方案。

### 12.2 高风险动作收口

明确不开放以下后台操作：

- 直接改支付结果
- 直接改出票结果
- 直接改订单核心状态
- 直接伪造退款完成

### 12.3 审计要求

以下操作必须记日志：

- 登录/登出
- 创建/编辑/发布/下架活动
- 新增订单备注
- 标记异常单
- 审核/驳回/发起退款
- 创建/禁用账号

6. 运营可添加备注和标记异常
7. 用户发起退款后后台可审核和处理

## 15. 最终结论

本次后台 MVP 将在保留现有技术底盘的前提下，升级为面向内部团队的真实运营工作台。

核心特征如下：

- 有正式后台登录和角色
- 有概览页
- 以活动上架为第一主流程
- 支持“区域票”而非“具体座位选座”
- 订单中心承担问题定位和跟进职责
- 退款中心承担标准处理队列职责
- 账号管理保持最小能力
- 所有关键动作可审计

该方案可以在当前项目阶段快速落地，又为后续扩展成完整票务后台保留清晰演进路径。
