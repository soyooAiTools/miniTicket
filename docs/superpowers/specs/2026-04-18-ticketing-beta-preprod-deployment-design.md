# 票务内测预发部署方案

## 背景

当前授权票务项目已经完成了核心业务链路代码、运行手册和内测值守文档，但还没有真正完成预发环境部署。当前仓库状态具备以下特点：

1. 代码已覆盖小程序、后台、API、数据库持久化、支付回调、出票回调、退款回调等主要能力。
2. 当前本机没有正在运行的 API、后台或负载控制服务。
3. 仓库中已有 [docker-compose.yml](../../../docker-compose.yml)，但目前只覆盖 `PostgreSQL` 和 `Redis`，尚未覆盖 API、Nginx、后台静态站点发布等内容。
4. 当前 CI 仅覆盖 `lint + test`，尚未形成正式的部署链路。
5. 当前后台默认 API 地址仍然写死为 `http://localhost:3000/api`，如果直接上线，会导致浏览器请求用户本机而不是服务器。

因此，这一轮的目标不是继续扩展业务功能，而是先把项目推进到“可部署、可联调、可回滚”的预发状态。

## 目标

本方案的目标是为 `B 阶段内测` 提供一套适合当前阶段的预发部署架构，并为后续脚本化部署做好清晰边界。

需要达到的结果是：

1. 预发环境可以稳定承载 `1 场真实演出` 的联调与小范围公开售卖。
2. API、后台、数据库、Redis、HTTPS、微信支付回调、上游回调都能在一台预发机上稳定跑通。
3. 部署方式足够简单，方便当前阶段快速迭代和排错。
4. 具备基础的发布、回滚、健康检查和备份能力。
5. 后续可以自然扩展为脚本化、一键化发布，而不需要推倒重做。

## 非目标

本方案当前明确不做以下事情：

1. 不直接设计正式生产环境的多机高可用架构。
2. 不在这一轮引入 Kubernetes、服务网格、复杂容器编排或云原生网关。
3. 不在这一轮拆出独立对象存储、独立日志平台或托管告警平台。
4. 不追求跨地域容灾或多活。
5. 不把预发环境设计成正式生产长期承载高并发的最终形态。

## 方案结论

预发环境采用：

`单台 Linux 云服务器 + 宿主机 Nginx + Docker Compose`

文中出现的 `beta.yourdomain.com` 仅作为示例域名，实际实施时应替换成你们持有并已完成解析的预发域名。

其中：

1. 宿主机负责 `Nginx + HTTPS + 静态后台站点入口`
2. Docker Compose 负责长期运行的服务容器：
   `api`
   `postgres`
   `redis`
3. 小程序不部署在服务器上，而是构建后上传到微信后台，运行时请求预发 API

这是当前阶段最合适的方案，因为它同时满足：

1. 部署链路足够简单
2. 同域路由可避免 CORS 问题
3. 便于快速回滚
4. 便于后续直接落成脚本

## 架构总览

预发环境的推荐结构如下：

1. `beta.yourdomain.com`
   统一作为预发域名
2. 宿主机 `Nginx`
   对外监听 `80/443`
3. `Nginx /api/*`
   反向代理到 Docker Compose 中的 `api` 容器
4. `Nginx /`
   直接托管后台构建产物 `admin dist`
5. Docker Compose 中运行：
   `api`
   `postgres`
   `redis`
6. 小程序运行时调用：
   `https://beta.yourdomain.com/api`

这一结构的核心原则是：

1. 后台和 API 保持同域
2. 回调地址保持稳定
3. 静态资源和业务 API 职责分离
4. 数据层只保留数据库和缓存两类常驻服务

## 域名与 HTTPS 设计

预发阶段建议只使用一个主域名：

`https://beta.yourdomain.com`

对外路由如下：

1. `https://beta.yourdomain.com/`
   提供后台页面静态资源
2. `https://beta.yourdomain.com/api/*`
   提供 API 能力
3. `https://beta.yourdomain.com/.well-known/acme-challenge/*`
   用于证书申请和续期

回调地址统一使用该域名下的 API 路由：

1. 微信支付回调：
   `https://beta.yourdomain.com/api/payments/wechat/callback`
2. 上游出票回调：
   `https://beta.yourdomain.com/api/fulfillment/vendor-callback-issued`
3. 上游退款回调：
   `https://beta.yourdomain.com/api/refunds/vendor-callback`

这样设计的好处是：

1. 所有外部回调地址统一、稳定、易排查
2. 后台页面与 API 同域，无需额外 CORS 处理
3. 更容易做 HTTPS 证书维护和统一入口监控

## Nginx 设计

宿主机 Nginx 负责三件事：

1. 提供 HTTPS 和 HTTP 到 HTTPS 跳转
2. 提供后台静态站点入口
3. 反向代理 `/api/*` 到 API 容器

Nginx 需要满足的关键行为：

1. `/api/` 走反向代理，不走静态文件目录
2. `/` 走后台构建产物目录
3. 后台前端路由刷新不能 404，因此要支持 `try_files ... /index.html`
4. 需要预留证书续期路径
5. 建议开启基础访问日志和错误日志

这一层不建议容器化，原因是：

1. 证书续期在宿主机处理更直接
2. 静态站点路径切换更方便
3. 发布和回滚时，宿主机 Nginx 只需要指向新的 release 目录

## Docker Compose 服务设计

预发阶段的 Compose 仅保留三个常驻服务：

1. `api`
2. `postgres`
3. `redis`

不建议把以下内容做成常驻容器：

1. `admin`
   因为后台是静态站点，更适合由 Nginx 直接托管构建产物
2. 数据库迁移任务
   迁移更适合在部署脚本中以一次性命令执行，而不是长期驻留容器
3. 小程序
   小程序不在服务器上运行

Compose 需要满足的基本规则：

1. `api` 依赖 `postgres` 和 `redis`
2. `postgres` 和 `redis` 都必须挂载持久化卷
3. `api` 通过环境变量读取数据库、缓存、微信支付和上游配置
4. 宿主机不直接暴露数据库和 Redis 给公网

## 服务器目录结构

推荐服务器根目录统一为：

`/srv/ticketing-beta`

下面拆成四层：

1. `releases/`
   每次发布生成一个独立目录，例如：
   `2026-04-18-001/`
2. `shared/`
   放不随发布版本切换的内容，例如：
   `.env`
   `nginx` 片段
   备份目录
   长期日志目录
3. `current/`
   软链接，始终指向当前生效版本
4. `ops/`
   放部署、回滚、健康检查、备份脚本

采用这种结构的原因是：

1. 部署时不覆盖旧文件
2. 回滚时只需要切软链接和重启相关服务
3. 版本目录天然可追溯

## 发布策略

预发发布采用“构建产物上传 + release 切换”的方式，不直接在服务器工作目录里做覆盖式发布。

推荐链路如下：

1. 在本地或 CI 产出构建结果
2. 将构建结果上传到服务器新的 release 目录
3. 在服务器执行数据库迁移
4. 通过 Docker Compose 启动或更新 `api` 容器
5. 将 `current` 软链接切换到新的 release
6. 验证通过后生效

构建物归属建议如下：

1. `admin`
   build 出 `dist` 后，放到 release 目录中，由 Nginx 直接读取
2. `api`
   build 出运行产物，并配合 Dockerfile 构建镜像或容器运行目录
3. `miniapp`
   构建后交付微信平台，不进入服务器常驻运行

## 回滚策略

回滚必须作为一等能力单独设计。

回滚动作建议固定为：

1. 找到上一版 release 目录
2. 将 `current` 指回上一版
3. 重新加载 Nginx 静态路径
4. 重新启动或回退 API 容器
5. 执行健康检查

需要明确的是：

1. 数据库迁移如果是破坏性变更，不能假设回滚永远无损
2. 预发阶段数据库迁移要尽量采用向后兼容策略
3. 回滚前必须清楚当前版本是否已经写入新结构数据

## 脚本拆分方案

部署自动化建议拆成五个脚本，而不是一个大而全脚本：

1. `prepare-server.sh`
   作用：
   服务器首次初始化
   安装 Docker、Compose、Nginx、Node 基础工具
   创建目录结构
2. `deploy-beta.sh`
   作用：
   常规发版
   上传 release
   切换 `current`
   执行迁移
   启动容器
   做健康检查
3. `rollback-beta.sh`
   作用：
   回滚到上一版 release
4. `check-beta.sh`
   作用：
   执行健康检查
   验证 Nginx、API、数据库依赖和关键业务入口
5. `backup-beta.sh`
   作用：
   备份数据库和关键配置

这样拆分的原因是：

1. 首次装机和日常发版不是同一类动作
2. 回滚必须可独立执行
3. 健康检查脚本在值守期也能单独复用

## 环境变量分层

预发环境建议拆成三层变量来源：

### 1. 服务器运行时环境

位置建议：

`/srv/ticketing-beta/shared/.env`

主要用于 API 容器读取，包括：

1. `PORT`
2. `DATABASE_URL`
3. `REDIS_URL`
4. `ADMIN_API_SECRET`
5. `VIEWER_ID_CARD_KEY`
6. 微信支付相关配置
7. 上游接口相关配置
8. 其他 API 运行时密钥

### 2. 后台运行环境

后台建议不依赖复杂的构建时环境注入，而是尽量依赖同域路径。

理想行为是：

1. 后台默认请求 `/api`
2. 只有在特殊场景下，才通过浏览器 localStorage 覆盖 API 地址

这样可以避免：

1. 线上后台默认请求用户本机 `localhost`
2. 不同环境的后台构建物需要频繁重编译

### 3. 小程序构建时环境

小程序构建时明确设置：

`TARO_APP_API_BASE_URL=https://beta.yourdomain.com/api`

这样小程序在预发环境里始终请求正确的 API 域名。

## 当前必须调整的代码点

为了让预发部署方案真正成立，这一轮部署实施前至少需要补齐以下改动：

1. 后台默认 API 地址不能继续写死 `http://localhost:3000/api`
   推荐改为：
   `/api`
   或
   `window.location.origin + '/api'`
2. API 需要补充明确的生产构建和启动方式
   当前仓库只有 `dev` 和 `test` 类脚本，不足以支撑标准化部署
3. 需要补充 API Dockerfile
4. 需要补充预发用 Compose 编排文件
5. 需要补充 Nginx 配置模板
6. 需要补充部署、回滚、健康检查和备份脚本

## 安全与暴露策略

预发环境虽然不是生产，但也不能裸跑：

1. 域名必须启用 HTTPS
2. 数据库和 Redis 只允许内网或本机访问
3. `ADMIN_API_SECRET` 不能写入仓库
4. `VENDOR_CALLBACK_SECRET` 不能写入仓库
5. 微信支付相关私钥和 API 密钥不能写入仓库
6. 宿主机建议至少开启基础防火墙，仅开放 `80/443`

## 运维最小能力

预发环境建议具备以下最小运维能力：

1. 能看 Nginx 访问日志和错误日志
2. 能看 API 容器日志
3. 能看 Docker Compose 服务状态
4. 能快速执行健康检查脚本
5. 能快速执行回滚脚本
6. 能在开卖前做一次数据库备份

## 验收标准

本方案完成后的验收标准应为：

1. 通过一个统一域名可以访问后台和 API
2. 后台默认不再请求 `localhost`
3. 小程序能请求预发 API
4. 微信支付回调能到达 API
5. 上游出票/退款回调能到达 API
6. 数据库和 Redis 通过 Compose 正常运行
7. 能执行一键部署
8. 能执行一键健康检查
9. 能执行一键回滚

## 推荐实施顺序

建议下一步按这个顺序实施：

1. 修正后台默认 API 地址策略
2. 给 API 补齐生产构建和启动脚本
3. 编写 API Dockerfile
4. 编写预发 Compose 文件
5. 编写 Nginx 配置模板
6. 编写服务器初始化脚本
7. 编写部署、回滚、健康检查和备份脚本
8. 在一台 Linux 预发机上实际跑通
9. 用小程序和后台进行真实联调

这样可以先把部署基础打牢，再去做环境实跑和值守验证。
