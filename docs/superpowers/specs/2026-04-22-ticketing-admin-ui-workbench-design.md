# 票务后台工作台 UI 设计

## 1. 背景

当前后台 MVP 的后端主骨架已经进入可实现状态：

- 已完成后台登录、账号管理、审计日志、概览汇总、活动管理、区域票档等后端基础
- 订单与退款后台处理能力已进入实现阶段
- 现有 `apps/admin` 仍然停留在英文表格页和演示型页面，无法承载“正式运营工作台”的体验目标

本次设计聚焦后台前端 UI，不重新定义业务范围，而是在既有 MVP 业务范围内，把后台界面升级为：

- 中文化
- 高密度但不混乱
- 官方高级、克制、偏值守型工作台
- 真正适合运营日常处理事务

## 2. 目标与非目标

### 2.1 目标

本次 UI 设计的目标如下：

- 将后台改造成高密度处理台，而不是通用模板后台
- 保持“概览总控页 -> 列表页 -> 独立详情页”的清晰处理流
- 将活动管理做成真正的上架主流程页面
- 将订单和退款做成可持续值守的高频处理页面
- 完成全局中文化，移除当前英文标题、英文状态和英文操作文案
- 在不牺牲实现效率的前提下，建立统一的后台视觉系统

### 2.2 非目标

本次 UI 设计不包含以下内容：

- 重新定义后台业务边界
- 引入新的一级模块
- 设计独立的履约顶层页面
- 用 `frontend-slides` 直接生成生产级后台页面代码
- 做成重图表、重展示的大屏型后台

## 3. 核心设计结论

通过本轮对齐，后台 UI 的关键决策如下：

- 整体方向：`高密度处理台`
- 默认落点：`概览总控页`
- 处理方式：`列表页 -> 独立详情页`
- 首屏优先级：`待办优先`
- 实现策略：`自定义工作台外壳 + Ant Design 业务组件`

上述结论意味着后台不会走“极简稀疏看板”路线，也不会走“纯组件库默认皮肤”路线，而是走一个更适合运营值守的中高密度工作台方向。

## 4. 视觉原则

### 4.1 视觉气质

整套后台采用以下视觉关键词：

- 官方高级
- 冷白灰浅色基调
- 克制、专业、偏业务
- 信息多但秩序明确

不追求花哨动效，不做消费级产品那种情绪化界面，而是强调可读性、稳定性和判断效率。

### 4.2 语言与文案

页面语言统一为中文：

- 页面标题中文
- 导航中文
- 按钮中文
- 状态中文
- 表格列名中文
- 操作反馈中文

同时去掉解释型长文案，不使用“这页是做什么的”这类介绍语。所有文案只服务于当前处理动作。

### 4.3 组件策略

采用“外壳自定义 + 业务组件复用”的策略：

- 自定义：导航、页面头部、概览卡片、状态区块、待办结构、详情页信息分区
- 复用 Ant Design：表格、表单、日期、弹窗、确认框、分页、消息提示

这样既能获得正式产品感，也能控制实现成本和维护复杂度。

## 5. 信息架构

后台一级导航固定为：

- `概览`
- `活动`
- `订单`
- `退款`
- `账号`

补充页面：

- `登录`
- `订单详情`
- `退款详情`
- `活动编辑`

不保留一级导航：

- `履约`

履约信息折叠进订单详情页，由订单详情承载支付、履约、退款、备注和异常标记，避免运营在多个页面之间来回切换。

## 6. 全局布局

整体布局采用三层结构：

1. 左侧固定导航
2. 顶部工具栏
3. 主工作区

### 6.1 左侧导航

左侧导航承担模块切换和当前位置提示，不承担复杂操作。要求：

- 宽度稳定
- 当前模块高亮清晰
- 图标少而克制
- 不堆叠说明文字

### 6.2 顶部工具栏

顶部工具栏承担全局操作和当前上下文：

- 页面标题
- 当前账号与角色
- 退出登录
- 页面级主操作入口

顶部不放欢迎语，不放装饰性统计，不做“品牌介绍头图”。

### 6.3 主工作区

主工作区是整套后台的重心，遵循：

- 默认密度偏高
- 通过分区和卡片控制节奏
- 状态信息优先于装饰
- 重要操作靠近数据本身

## 7. 概览总控页

概览页是默认首页，定位不是展示型数据大盘，而是待办优先的中控页。

页面从上到下分四层：

### 7.1 今日待处理摘要

顶层展示高优先级待办卡，例如：

- 待审核退款
- 异常订单
- 今日开售活动
- 需人工跟进事项

这些卡片必须可点击，点击后直接进入对应筛选结果。

### 7.2 重点处理队列

主区域左侧展示今日最需要先处理的事务：

- 退款待审核
- 异常订单
- 即将开售活动
- 用户争议单

这里更像任务队列，而不是纯数字汇总。

### 7.3 运营概况

右侧展示压缩版状态信息，例如：

- 售卖中活动数
- 待开售活动数
- 今日新增订单
- 今日退款申请数
- 最近后台操作

这部分是辅助认知，不应抢过待办主线。

### 7.4 风险提醒与最近操作

底部用于承接：

- 最近操作日志
- 库存偏低提醒
- 活动资料不完整提醒
- 退款积压提醒

概览页不以图表优先，而以“快速进入处理状态”为优先。

## 8. 活动管理

活动管理是后台第一优先业务页，采用“列表页 + 编辑页”结构。

### 8.1 活动列表页

列表页顶部包含：

- 活动名搜索
- 城市筛选
- 状态筛选
- 创建活动按钮

列表每行重点显示：

- 活动名称
- 城市 / 场馆
- 开售时间
- 当前状态
- 最低价
- 最近更新时间

行内操作保持克制：

- 编辑
- 发布 / 下架
- 查看详情

### 8.2 活动编辑页

活动编辑页按业务区块组织，而不是长表单直排：

- 基本信息
- 销售规则
- 区域票档
- 售后规则

页面顶部固定主操作：

- 保存草稿
- 发布
- 下架

### 8.3 区域票档

活动编辑页中，区域票档是核心模块。第一版明确支持“区域票”，不支持座位图选座。

每个区域票档至少包含：

- 区域名称
- 价格
- 库存
- 每单限购
- 是否实名
- 是否可退
- 退款截止时间
- 排序

编辑体验要强调结构化录入，而不是把区域票档塞进普通文本表单。

## 9. 订单中心

订单中心用于定位问题和协同处理，不用于高风险状态篡改。

### 9.1 订单列表页

顶部支持：

- 订单号搜索
- 用户搜索
- 活动搜索
- 状态筛选
- 异常单筛选

列表中每行重点显示：

- 订单号
- 活动名 / 场次
- 区域票档
- 订单状态
- 支付状态
- 退款状态
- 下单时间

### 9.2 订单详情页

订单详情页承载以下区块：

- 基本订单信息
- 用户与观演人信息
- 区域票档与金额信息
- 支付信息
- 履约记录
- 退款记录
- 内部备注
- 异常标记

允许的运营动作：

- 添加内部备注
- 标记异常单
- 查看履约记录
- 跳转到退款处理

不开放：

- 手工改支付成功
- 手工改出票成功
- 任意改订单主状态

## 10. 退款中心

退款中心用于标准化审核和处理。

### 10.1 退款列表页

顶部支持：

- 退款单号搜索
- 订单号搜索
- 活动搜索
- 状态筛选

列表中每行重点显示：

- 退款单号
- 关联订单
- 活动名
- 当前状态
- 申请金额
- 申请时间
- 退款原因

### 10.2 退款详情页

退款详情页承载：

- 订单摘要
- 用户信息
- 区域票档信息
- 可退金额与申请金额
- 原因与历史记录
- 审核信息
- 处理备注

允许的运营动作：

- 审核通过
- 驳回并填写原因
- 发起退款
- 添加备注

订单中心与退款中心职责清晰分工：

- 订单中心负责发现问题、定位问题、记录备注、查看履约
- 退款中心负责审核与处理退款

## 11. 登录与账号管理

### 11.1 登录页

登录页保持极简，只保留：

- 邮箱
- 密码
- 登录按钮

登录成功后默认进入 `概览` 页。

### 11.2 账号管理

账号管理页只对超级管理员开放，第一版支持：

- 账号列表
- 创建账号
- 启用 / 停用账号
- 角色切换

角色仅保留：

- `超级管理员`
- `运营`

不做细粒度权限树。

## 12. 状态系统

后台需要统一状态表现，不允许各页颜色和文案漂移。

第一版至少统一以下类别：

- 待处理
- 已完成
- 异常
- 审核中
- 驳回
- 售卖中
- 待开售
- 已下架

状态标签用于辅助判断，不抢占主内容权重。

## 13. `frontend-slides` 的使用边界

`frontend-slides` 只用于前置视觉探索，不直接作为后台实现方案。

具体边界如下：

- 用它快速生成后台方向稿和视觉探索稿
- 用它帮助对齐密度、布局、气质和层级
- 不将生成的 HTML 直接当成 `apps/admin` 页面代码

真正的后台实现仍然落在：

- React
- React Router
- Ant Design
- 现有请求与认证体系

## 14. 实现建议

前端实现建议按以下顺序推进：

1. 搭建统一后台外壳、认证上下文、路由守卫
2. 落地概览总控页
3. 落地活动列表与活动编辑页
4. 落地订单列表与订单详情页
5. 落地退款列表与退款详情页
6. 落地账号页
7. 全量中文化和视觉统一收口

## 15. 验收标准

本次 UI 设计落地后，应满足以下标准：

- 后台整体为全中文
- 默认进入待办优先的概览页
- 订单和退款采用列表页进入独立详情页
- 活动编辑支持结构化区域票档配置
- 履约不再作为一级导航存在
- 视觉效果明显区别于默认 Ant Design 后台模板
- 运营可以在不阅读说明文案的前提下理解页面用途并完成处理动作
