# 票务 B 阶段内测联合运行手册

## 目的

这份运行手册把授权票务平台的内测工作拆成一条连续执行链，覆盖两个紧密相连的阶段：

1. `预发联调`
2. `首场真实开卖值守`

目标不是做一套宏观上线治理文档，而是帮助一个小团队把 `1 场真实演出` 从最终联调顺利带到第一次受控公开开卖，并且在支付、出票、退款、客服沟通上不失控。

## 适用范围

这份手册默认基于当前已经确认的内测边界：

1. 只卖 `1` 场真实演出
2. 使用真实微信小程序登录
3. 使用真实微信支付
4. 出票走上游接口提交和回传
5. 退款走上游接口提交和回传
6. 小范围真实公开售卖
7. 只允许异常场景走人工兜底，主链路不能靠人工完成

## 当前实现边界

团队在执行时需要明确现在产品的真实边界：

1. 公开浏览接口只返回已发布演出。`published=false` 会让该演出从 `GET /api/catalog/events` 和 `GET /api/catalog/events/:eventId` 中消失。
2. `saleStatus` 和 `refundEntryEnabled` 目前是可见的运营控制项和用户端状态控制项，但还不是“后端硬熔断”，不能单靠这两个字段就完全阻止草稿单创建或退款提交。
3. 微信支付回调当前依赖平台证书、序列号校验和 API v3 解密流程验真。`WECHAT_CALLBACK_SECRET` 只是预留字段，当前回调路由并不会使用它做共享密钥校验。
4. 上游出票回调和退款回调都通过 `x-vendor-callback-secret` 保护。
5. 管理后台接口通过 `x-admin-secret` 保护。
6. 这份手册默认内测期间主要依赖管理后台和部署日志作为事实来源，不默认已经有成熟的独立告警平台。

## 值守角色

首场真实开卖建议至少明确 4 个角色。团队很小时可以一人兼多岗，但职责边界必须清楚：

1. `总指挥`
   负责开卖 go/no-go 决策、销售状态切换和升级处理拍板。
2. `后端值守`
   负责 API 健康、支付/上游回调可见性、日志排查和关键接口可用性。
3. `运营值守`
   负责后台 `/events`、`/orders`、`/fulfillment`、`/refunds` 页面，以及异常订单归属和队列状态。
4. `客服值守`
   负责用户口径、异常单升级标签和对外沟通。

当前代码里实际使用的是这些路由和操作入口：

1. `POST /api/auth/wechat/login`
2. `POST /api/orders/draft`
3. `POST /api/payments/wechat/intent`
4. `POST /api/payments/wechat/callback`
5. `POST /api/fulfillment/vendor-callback-issued`
6. `POST /api/refunds/request`
7. `POST /api/refunds/vendor-callback`
8. `GET /api/catalog/admin/events`
9. `PATCH /api/catalog/admin/events/:eventId`
10. `GET /api/orders/admin`
11. `GET /api/fulfillment/admin`
12. `GET /api/refunds/admin`
13. 后台页面 `/events`、`/orders`、`/fulfillment`、`/refunds`

## 第一阶段：预发联调

### 1. 环境冻结

外部联调开始前，后端值守需要先确认预发环境和当前仓库约定一致：

1. 确认 `.env` 已按 [.env.example](../../../.env.example) 补齐
2. 确认预发 API 地址可访问
3. 确认后台可通过浏览器 localStorage 指向预发 API：
   `ticketing.admin.apiSecret`
   `ticketing.admin.apiBaseUrl`
4. 确认数据库迁移已经执行
5. 确认预发环境使用的是正确的上游地址，而不是旧的 sandbox 地址
6. 确认微信支付通知地址指向 `POST /api/payments/wechat/callback`
7. 确认上游回调方能正确携带 `x-vendor-callback-secret`

### 2. 后台启动核验

正式有用户流量前，运营值守需要先把后台核通：

1. 打开后台应用
2. 设置浏览器 localStorage：

```js
localStorage.setItem('ticketing.admin.apiSecret', '<ADMIN_API_SECRET>');
localStorage.setItem('ticketing.admin.apiBaseUrl', 'https://beta.example.com/api');
```

3. 刷新后台页面
4. 打开 `/events` 并确认目标演出存在
5. 打开 `/orders`、`/fulfillment`、`/refunds` 并确认没有鉴权报错

### 3. 演出配置核验

总指挥和运营值守一起确认演出记录本身没有配置错误：

1. 演出标题、城市、场馆、最低价正确
2. 至少存在一个场次
3. 票档库存正确，且票型为预期的 `E_TICKET` 或 `PAPER_TICKET`
4. `published=true`
5. `saleStatus` 符合当前阶段
6. 如需开放退款入口，则 `refundEntryEnabled=true`
7. 小程序内的购票规则、退票规则、实名说明和既定政策一致

### 4. 微信登录联调

用一个真实测试账号验证：

1. 小程序能正常执行 `wx.login`
2. `POST /api/auth/wechat/login` 调用成功
3. 返回的 session 能让用户创建观演人、查看订单
4. 重开小程序后，仍然能以同一个客户身份看到自己的订单历史

只有当“订单归属在重复打开小程序后仍然稳定”时，这一项才算通过。

### 5. 支付链路联调

跑一次完整的草稿单和支付流程：

1. 创建一个观演人
2. 从目标演出选择一个票档创建草稿单
3. 调用 `POST /api/payments/wechat/intent`
4. 完成一笔真实微信支付
5. 确认 `POST /api/payments/wechat/callback` 确实到达 API
6. 确认订单离开草稿态，进入已支付或待履约确认态
7. 确认订单详情页不会在刚支付完时错误显示为“已出票”

这一项通过的标准是：支付成功后，后端状态和用户端状态都发生了可见且正确的变化。

### 6. 出票链路联调

跑一次真实出票回传流程：

1. 确认已支付订单已经提交到上游
2. 确认订单出现在后台 `/fulfillment`
3. 触发或等待一次真实的 `POST /api/fulfillment/vendor-callback-issued`
4. 确认回调携带了 `x-vendor-callback-secret`
5. 确认订单在后台和小程序订单详情中都进入“已出票”
6. 确认票码和上游事件标识在预期的运营视图中可见

如果预发阶段拿不到真实出票回传，就不要进入第一次真实公开开卖。

### 7. 退款链路联调

选一笔受控订单跑一次退款提交和退款回传：

1. 在小程序里打开一笔已支付或已出票订单
2. 通过 `POST /api/refunds/request` 提交退款
3. 确认退款请求出现在后台 `/refunds`
4. 确认退款请求已经发给上游
5. 触发或等待一次真实的 `POST /api/refunds/vendor-callback`
6. 确认回调携带了 `x-vendor-callback-secret`
7. 确认订单和退款请求都进入最终退款完成态

### 8. 规则核验

客服值守要把规则敏感项逐条过一遍：

1. 电子票显示 `演出开始前三天` 的时间规则
2. 纸质票显示 `演出开始前七天` 的时间规则
3. 订单详情和售后流程清楚说明“下单后、出票前不能修改观演人信息”
4. `USER_IDENTITY_ERROR` 且 `daysBeforeStart <= 3` 的退款计算会扣除 `20%` 服务费
5. 其他退款原因不会误扣这笔服务费

### 9. 彩排矩阵

只有把下面这些场景都跑过，预发联调才算完整：

1. 正常路径：登录 -> 草稿单 -> 支付 -> 出票
2. 支付发起后、回调前失败
3. 支付成功但进入待履约确认
4. 上游出票失败或超时
5. 退款申请成功并收到退款成功回调
6. 退款回调延迟或超时
7. 临演前三天内身份信息错误退款计算
8. 一笔异常订单在后台可见且有明确负责人

### 10. Go / No-Go 决策门

总指挥只有在以下 5 项全部为真时，才应该放行公开售卖：

1. 登录、支付、出票、退款四条链路都至少完成过一次真实联调
2. 后台页面能正常连到预发 API
3. 至少有一条异常单路径被团队完整演练过
4. 客服已经准备好“待出票、出票失败、退款中、退款失败、身份信息错误”的统一口径
5. 演出在小程序里可见且展示正确

其中任意一项不满足，都应该推迟第一次真实公开开卖。

## 第二阶段：首场真实开卖值守

### 1. T-60 分钟

团队需要先拉起指挥通道，并确认：

1. 总指挥、后端值守、运营值守、客服值守都在线
2. 后台页面已打开且已完成鉴权
3. 操作人员设备能访问 API 和后台地址
4. 目标演出的 `published`、`saleStatus`、`refundEntryEnabled` 状态符合预期
5. 当前上游沟通通道可用

### 2. T-30 分钟

后端值守和运营值守需要做一次轻量 smoke check：

1. `GET /api/catalog/events` 能返回目标演出
2. 小程序演出详情页能正常打开
3. 后台 `/orders`、`/fulfillment`、`/refunds` 能正常打开
4. 再跑一次新登录，确认仍然成功
5. 后台没有遗留的退款或履约积压，避免污染本场视图

### 3. T-10 分钟

总指挥需要做最终开卖确认：

1. 确认 `published=true`
2. 如果现在要开始售卖，确认 `saleStatus=ON_SALE`
3. 确认退款入口策略和用户侧展示一致
4. 确认客服值守已经准备好统一话术

### 4. 开卖窗口监控

从开卖时刻到第一小时稳定为止，需要持续看这些队列：

1. 小程序浏览、下单、支付反馈
2. 后台 `/orders` 中的已支付单和异常单
3. 后台 `/fulfillment` 中的已支付未出票单
4. 后台 `/refunds` 中的早期退款请求
5. 部署日志中的微信回调错误
6. 部署日志中的上游提交或上游回调错误

第一小时内，运营值守至少每 `10` 到 `15` 分钟向总指挥同步一次，即使只是“当前稳定”也要播报。

### 5. 第一小时稳定标志

只有以下 6 项都出现时，才可以认为第一小时进入稳定状态：

1. 新用户可以登录
2. 新草稿单可以创建
3. 支付可以完成
4. 支付成功后能进入履约处理
5. 至少有一笔订单进入用户可见的“已出票”
6. 后台页面数据持续可读且刷新正常

### 6. 客服值守口径

客服值守对外回复时要坚持这几个锚点：

1. `支付成功但还未出票`
   只能说“支付已完成，出票仍在确认中”，不能直接承诺已经出票。
2. `出票失败或出票延迟`
   只能说“订单已转人工跟进，团队正在核查上游结果”。
3. `退款处理中`
   只能说“退款请求已受理，正在等待上游确认”。
4. `临演前三天身份信息错误`
   要明确说明会按已公布规则扣除 `20%` 服务费。

### 7. T+90 分钟复盘点

总指挥需要暂停一下，看清楚：

1. 已支付订单数量
2. 已出票订单数量
3. 仍待出票订单数量
4. 退款申请数量
5. 用户投诉或未解决客服会话数量

如果待出票队列或退款异常队列增长速度已经超出团队可解释范围，就不要继续扩大流量暴露。

## 第三阶段：异常处理剧本

### 1. 已支付但还没有出票结果

当用户支付完成，但订单仍在等出票时，执行：

1. 确认订单出现在后台 `/orders`
2. 确认同一订单出现在后台 `/fulfillment`
3. 检查 `POST /api/payments/wechat/callback` 对应日志
4. 检查该订单的上游提交日志
5. 在拿到出票结果前，绝不能告诉用户“已经出票”
6. 如果支付回调成功后超过 `15` 分钟仍然没有出票进展，就转人工跟进并记录负责人

### 2. 上游出票失败

当上游明确失败，或长时间不给出票确认时，执行：

1. 记录订单 ID
2. 记录用户侧当前订单状态
3. 记录已返回的上游事件标识或票码（如果有）
4. 决定下一步是继续人工催出票，还是进入退款路径
5. 在运营记录里写清楚负责人
6. 给客服一个单一明确的回复路径和下次更新时间

### 3. 重复回调或状态不一致

当支付或上游回调重复、甚至看起来互相矛盾时，执行：

1. 先确认订单当前状态是否已经终态
2. 记录重复回调的时间和 payload 标识
3. 确认系统没有把终态订单回滚
4. 对外口径要始终以“最终确认状态”为准，不要跟着噪音回调摇摆
5. 如果后台和小程序订单详情状态不一致，立即升级处理

### 4. 用户身份信息错误

当观演人信息有误时，执行：

1. 判断订单是否仍处于出票前
2. 判断问题是否发生在演出前最后 `3` 天内
3. 判断原因是否属于 `USER_IDENTITY_ERROR`
4. 如果 3 个条件都成立，则按规则扣除 `20%` 服务费，并把扣费依据写进 case 备注
5. 如果任意一个条件不成立，就不要自动套用这条晚期信息错误扣费规则

### 5. 退款超时或退款失败

当退款没有顺利结束时，执行：

1. 确认退款请求出现在后台 `/refunds`
2. 记录退款单号
3. 检查上游退款提交日志
4. 检查上游退款回调日志
5. 在彻底解决前，这笔单必须留在“处理中”或“异常”队列里，不能消失
6. 在拿到最终退款回调之前，不能告诉用户“退款已完成”

## 应急控制

总指挥可以使用这些控制项，但要清楚它们当前的实现边界：

1. 设 `published=false`，让演出从公开列表和详情页消失
2. 修改 `saleStatus`，改变用户可见的销售状态
3. 设 `refundEntryEnabled=false`，隐藏用户侧退款入口

但因为草稿单创建和退款提交还不能只靠这些字段被彻底拦住，所以遇到严重事故时，还必须同时做：

1. 立即停止推广和新流量导入
2. 人工盯住在途支付回调和上游回调
3. 检查后端日志，确认系统没有继续处理隐藏流量

## 值守期间必须保留的证据

每一笔异常单，至少要保留这些信息：

1. 订单 ID
2. 订单号
3. 客户 ID（如果可得）
4. 如果涉及支付，记录微信 `out_trade_no`
5. 如果支付回调已成功，记录微信支付交易号
6. 如果出票回调有返回，记录上游事件标识
7. 如果出票回调有返回，记录票码
8. 如果涉及退款，记录退款单号
9. 记录后台可见状态
10. 记录小程序订单详情可见状态
11. 记录当前负责人

## 值守结束复盘

首场值守窗口结束后，团队至少要记录：

1. 成功登录总数
2. 已支付订单总数
3. 已出票订单总数
4. 退款申请总数
5. 异常订单总数
6. 最常见的异常路径是什么
7. 是否曾经隐藏演出或修改销售状态
8. 扩大范围前必须先修的 3 个问题

下一步要不要扩大测试范围，应该基于这份复盘来决定，而不是只看成交金额。
