# 项目现状与下一步

最后更新：2026-04-21

## 1. 当前完成度

可以把当前状态理解为：

`核心骨架已齐 + 本地 mock 闭环可跑 + 小程序 UI 已升级 + 待真实环境验证`

### 1.1 已经完成的部分

#### 工程层

- 纯票务仓库已独立
- Monorepo 结构已经稳定
- API / Admin / Miniapp / Contracts 已打通
- 本地数据库迁移、seed、基础验证命令已补齐

#### 业务层

- 演出目录
- 演出详情
- 观演人
- 草稿单 / 订单
- 微信支付意图
- 出票回调
- 退款申请与退款回调

#### 小程序层

- 首页改成单列海报卡片流
- `Home / Events` 已收敛成一个主发现页模型
- 底部导航收敛为 `首页 / 订单 / 我的`
- 已补本地展示海报和 showcase 数据

#### 运维层

- 预发部署脚本已经入库
- Nginx 和 Compose 模板已经入库

## 2. 当前没有完成的部分

### 2.1 真环境接入

下面这些还没有完成真实落地验证：

- 真实小程序 AppID
- 真实微信登录
- 真实微信支付
- 真实上游出票回调
- 真实上游退款回调

### 2.2 服务器侧

- Linux 预发服务器还没有完成首轮真实部署
- 还没有完成预发环境全链路彩排

### 2.3 产品侧

- 当前 UI 已经脱离 demo 感，但还可以继续精修
- 后台管理端的视觉层还不是重点打磨对象
- 真实活动数据、真实票档、真实运营内容还没有完整装载

## 3. 当前已知风险

### 3.1 小程序当前偏开发态

当前为了便于演示和联调，小程序开发态默认依赖：

- `touristappid`
- `showcase` 数据
- 本地 mock 支付

这对本地开发友好，但交给团队后必须明确区分：

- 哪些是开发态便利能力
- 哪些是真实上线能力

### 3.2 预发部署脚本虽已写好，但还缺一次真实跑通

当前运维方案不是没有，而是“已设计、未完全实战验证”。

### 3.3 业务规则已经确定，但还要和真实运营流程再对齐一次

例如：

- 实名修改限制
- `20%` 服务费规则
- 出票 / 退款状态口径

这些已经有文档，但还需要在首场真实彩排中再次确认。

## 4. 建议下一步工作顺序

### 第一优先级

1. 完成本地接手和 smoke test
2. 替换真实小程序 AppID
3. 跑通真实微信登录

### 第二优先级

1. 跑通真实微信支付
2. 跑通真实上游出票 / 退款回调
3. 补一轮真实订单异常处理验证

### 第三优先级

1. 上 Linux 预发
2. 跑一次全链路彩排
3. 给运营做一次后台和值守培训

## 5. 建议的短期任务拆分

### Sprint 1：接手与真环境接入

- 接管仓库和依赖
- 跑通本地 mock 闭环
- 换真实微信 AppID
- 完成真实登录和真实支付接入

### Sprint 2：预发与彩排

- 部署预发
- 接真实 vendor
- 跑出票与退款回调
- 执行一场完整彩排

### Sprint 3：上线前收口

- 修真实彩排中暴露的问题
- 补运营侧 SOP 缺口
- 做上线前配置冻结与备份

## 6. 对开发团队的建议

当前最容易犯的错误不是“代码不会写”，而是：

- 把开发态开关当成正式能力
- 直接跳过本地联调去上预发
- 不了解业务规则就改订单状态逻辑
- 只看前台，不看后台异常处理路径

更稳的做法是：

1. 先把现有 mock 链路吃透
2. 再换真环境
3. 每接一层真能力就回归一次订单主链路
