# 需求摘要

最后更新：2026-04-21

## 1. 项目定位

`miniTicket` 是一个面向 C 端用户的官方票务平台，当前主要承载微信小程序票务场景。

它不是黄牛抢票工具，也不是单纯的信息展示页，而是一个完整票务闭环产品，目标是在用户侧完成：

- 演出浏览
- 场次与票档选择
- 实名观演人填写
- 在线支付
- 出票结果查看
- 售后与退款申请

## 2. 当前阶段

当前项目阶段不是正式大规模上线，而是 `B` 阶段内测准备。

这个阶段的目标是围绕 `1` 场真实演出，验证一条真实可运行的主链路：

`浏览 -> 实名下单 -> 支付 -> 出票 -> 售后/退款`

这个阶段不追求：

- 多场演出同时铺开
- 复杂选座
- 会员、积分、裂变、分销
- 没有人为兜底的全自动运营

## 3. 当前产品范围

当前仓库包含 3 个主要产品面：

### 3.1 微信小程序

面向用户的前台入口，当前主要页面包括：

- 首页 / 演出发现
- 演出详情
- 观演人管理
- 下单与支付
- 支付结果
- 订单列表
- 订单详情
- 售后申请
- 我的

### 3.2 API

当前 API 已经覆盖主要业务模块：

- 微信登录
- 演出目录与详情
- 观演人
- 草稿单与订单
- 微信支付意图与支付回调
- 上游出票提交与出票回调
- 退款申请与退款回调

### 3.3 后台管理端

当前后台主要用于运营和异常处理，重点是：

- 演出与票档控制
- 订单视图
- 履约视图
- 退款视图

## 4. 核心业务规则

### 4.1 实名规则

- 默认实名购票
- 下单后、出票前，不允许随意修改已绑定观演人
- 如果用户信息错误且发生在演出前 `3` 天内，导致无法录入，按规则扣除 `20%` 服务费

### 4.2 票务时效规则

- 电子票：最晚在演出开始前 `3` 天确认
- 纸质票：最晚在演出开始前 `7` 天确认

### 4.3 售后规则

- `支付成功` 不等于 `已出票`
- `退款申请已提交` 不等于 `退款已完成`
- 售后和客服必须以系统最终状态为准，不能口头超前承诺

## 5. 当前 UI/产品方向

当前小程序已经不是最初 demo 风格，而是按“官方高级、浅色质感、海报主导”的方向重做过一轮。

当前确认下来的视觉方向是：

- 官方票务平台感
- 冰白 / 极淡蓝灰
- 单列大海报卡片流
- 首页承载多场活动
- 底部导航收敛为 `首页 / 订单 / 我的`
- 全站尽量减少解释型文案

## 6. 当前非目标

当前这一阶段不建议开发团队主动扩 scope 到下面这些方向：

- 复杂选座引擎
- 多城市多上游并发铺开
- 社区内容能力
- 重营销活动系统
- 用户增长体系
- 过重的设计系统工程化改造

## 7. 接手团队最该抓住的重点

开发团队接手后，优先级建议是：

1. 保住现有本地 mock 闭环
2. 完成真实微信登录 / 真实微信支付接入
3. 完成真实上游出票 / 退款回调联调
4. 上 Linux 预发环境
5. 跑一场真实演出彩排
